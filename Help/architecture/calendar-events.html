Overview

13 partial files under Calendar/BeepCalendar.EventOperations.*.cs handle the complete event lifecycle plus the standalone CalendarEventEditor class. Every event mutation (add, update, delete) follows a three-phase pipeline: Pre-mutation (EventMutating with cancel support) → Execution (data change) → Post-mutation (EventMutated + history recording). An integrated undo/redo system records mutations on two stacks, and the conflict detection system checks scheduling overlap using configurable CalendarConflictPolicyMode.

CRUD Operations

All event mutations follow a strict three-phase pipeline:

  1. Pre-mutation — RaiseMutating(kind, source, proposed, original, isNew, out canceled)
    • Fires EventMutating event with CalendarEventMutationEventArgs
    • Handlers may cancel by setting e.Cancel = true (validation, permissions, business rules)
  2. Execution — The actual data change: add to Events list, update properties, or remove from list
  3. Post-mutation — RaiseMutated(kind, source, result, original, isNew, conflicts)
    • Records mutation in undo/redo history with full state snapshot
    • Fires EventMutated event for UI refresh
    • Updates view button states (undo/redo enabled)

Mutation Pipeline

// Cancel a delete based on business rules
cal.EventMutating += (s, e) =>
{
    if (e.Kind == CalendarEventMutationKind.Delete
        && e.Original.Category == "Locked")
    {
        e.Cancel = true;  // Prevent deletion of locked events
    }
};

// React to completed mutations
cal.EventMutated += (s, e) =>
    Console.WriteLine($"{e.Kind}: {e.Result.Title}");

// Create event request
cal.CreateEventRequested += (s, e) =>
{
    e.CreatedEvent = CalendarEventEditor.ShowNewEventDialog(
        e.RequestedDate, e.RequestedStart);
};

Undo/Redo History

Two stacks manage the complete history:

  • _undoStack — Stack of mutation records, each containing the full before/after state of the event
  • _redoStack — Stack of undone mutations for redo capability

    // Returns false if stack is empty
}

public bool RedoLastMutation()
{
    // Pop _redoStack, reapply forward, push back to _undoStack
}

// Convenience API
if (cal.CanUndo) cal.Undo();
if (cal.CanRedo) cal.Redo();

History is implemented across 5 files: History.cs (dispatcher), History.Internal.cs (stack management), History.Apply.cs / Apply.Upsert.cs / Apply.Remove.cs (reverse mutation logic). Each mutation record stores the complete event state before and after the change.

Conflict Detection

// Three policy modes
cal.ConflictPolicyMode = CalendarConflictPolicyMode.Strict;
// None:   No conflict checking
// Warn:   Detect and fire ConflictDetected, but don't block
// Strict: Detect and block overlapping events

cal.ConflictDetected += (s, e) =>
{
    Console.WriteLine($"Conflict: {e.Description}");
    foreach (var conflict in e.ConflictingEvents)
        Console.WriteLine($"  {conflict.Title} ({conflict.Start}-{conflict.End})");
};

The conflict detection system uses a CalendarConflictPolicy object (lazy-created from _conflictPolicyMode) to check scheduling overlap before commit. Overlapping events are returned as CalendarConflictEventArgs.

Categories & Resources

  • Categories — List<EventCategory>. Each category has Name, Color (for visual indicators), Icon path, and optional parent category for hierarchy. Events reference a category via CalendarEvent.Category.
  • Resources — List<CalendarResource> for room/equipment/person scheduling. Each resource has Name, Type, Capacity, Color, and optional Location.
  • PerformanceMetrics — CalendarPerformanceMetrics object (browsable=false) for tracking rendering time, interaction latency, and event operation timing.

Public Commands

5 command files expose the public command API: BeepCalendar.Commands.cs (main command definitions), Commands.Core.cs (execution engine), Commands.Helpers.cs (mutation context creation), Commands.DuplicateDetection.cs (dedup logic for command batching), Commands.MutationTracking.cs (mutation identity generation for auditing). Commands drive the public API indirectly — the calendar exposes user-facing methods that internally route through the command system for consistency.

Add Event with Validation

cal.EventMutating += (s, e) =>
{
    if (e.Proposed.End - e.Proposed.Start > TimeSpan.FromHours(8))
        e.Cancel = true;  // Reject events longer than 8 hours
};

cal.Events.Add(new CalendarEvent
{
    Title = "Design Review",
    Start = DateTime.Today.AddHours(10),
    End = DateTime.Today.AddHours(11),
    Location = "Room 204",
    Category = categories["Meeting"]
