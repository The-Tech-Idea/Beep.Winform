BeepDataRecord

A dynamic data-bound record editor that auto-generates form fields from any CLR object, with full edit tracking, validation, and IEditableObject support.

Overview

BeepDataRecord is a data-bound record editor that dynamically generates a form layout from any CLR object via reflection. It introspects the object's public properties and automatically creates the appropriate input controls — BeepTextBox for string/numeric types, BeepComboBox for enum types — arranged in a responsive TableLayoutPanel.

The control supports two-way data binding with built-in dirty tracking, original-value snapshots, per-field undo, and full IEditableObject implementation for integration with BindingSource and DataGridView. Validation runs automatically on every field change, highlighting required-but-empty fields in red.

Key Features

Auto Field Generation

Reflects over any CLR object's public properties and automatically instantiates the correct control type (BeepTextBox for primitives/strings, BeepComboBox for enums) via ControlExtensions.CreateFieldBasedOnCategory.

Two-Way Data Binding

Changes in the UI propagate back to the data object in real time. BindDataRecord pushes object values into fields; UpdateDataRecord pulls field values back to the object.

Edit Tracking & Undo

Tracks original values in a dictionary snapshot. Supports per-field undo via UndoField and full record undo via UndoChanges. Maintains a RecordStatus state machine (New / Updated / Unchanged).

IEditableObject Support

Full implementation of BeginEdit, CancelEdit, and EndEdit for seamless integration with BindingSource, DataGridView, and other data infrastructure.

Dual Orientation Layout

Switch between FieldOrientation.Horizontal (fields in columns) and FieldOrientation.Vertical (fields stacked in rows). Optional field prompt labels displayed above or beside each field.

Built-in Validation

Validates all fields on every value change. Required-but-empty fields turn red and show a tooltip. Fires RecordValidated with a boolean indicating overall validity.

Deep Copy Snapshots

Uses EntityHelper.DeepCopyUsingSerialize to create independent copies of data records for undo/rollback, ensuring no reference entanglement with the original object.

Full Theme Integration

Inherits BaseControl's theme system and propagates the current theme to all auto-generated child controls, ensuring consistent visual styling.

Layout & Orientation

BeepDataRecord uses an internal Panel containing a TableLayoutPanel for layout. The panel is sized to match the BaseControl.GetAdjustedContentRect() material area on every paint and resize event, ensuring proper alignment with the control's themed borders and shadows.

Horizontal Layout

When FieldOrientation = Horizontal, fields are laid out in a single row per layer. If ShowFieldPrompts is true, labels occupy row 0 and input controls occupy row 1; otherwise only one row is used. Column count equals FieldCount, with each column getting an equal percentage of the total width.

Vertical Layout

When FieldOrientation = Vertical, the layout switches to two columns (label + input) with each field pair occupying its own row. The label column is fixed at 150px with the input column filling remaining space. If ShowFieldPrompts is false, only one column is used — just the inputs, stacked vertically.

Empty State

When no fields exist, a centered BeepLabel displays "No fields available" in a single-cell layout.

Field Labels

Labels are generated by CreateFieldLabel, which checks FieldPrompts for custom text. If no custom text is found, the ComponentName (property name) is used with underscores replaced by spaces.

Edit Tracking & IEditableObject

The control maintains two independent snapshot mechanisms:

  • Deep Copy (_originalDataRecord): A serialization-based clone created at SetDataRecord time. Used by UndoChanges and IEditableObject.CancelEdit for full rollback.
  • Original Values Dictionary (_originalValues): A property-name-to-value map populated at SetDataRecord time. Used by UndoField for per-field rollback.

The RecordStatus state machine transitions as follows:

  • Calling SetDataRecord(row, isNew: true) → RecordStatus.New
  • Calling SetDataRecord(row, isNew: false) → RecordStatus.Unchanged
  • Any field value change → RecordStatus.Updated

The IEditableObject interface is fully wired: BeginEdit saves a snapshot, CancelEdit restores from it, and EndEdit finalizes the edit by clearing the snapshot and marking the record as Updated.

Theming

BeepDataRecord inherits the full material theming pipeline from BaseControl. The ApplyTheme override:

  1. Calls base.ApplyTheme() to handle border, shadow, background, and font tokens.
  2. Sets _mainPanel.BackColor and _tableLayout.BackColor to match the control's background.
  3. Iterates over all generated _fields, assigns the current Theme to each, and calls field.ApplyTheme().

The DrawContent override calls base.DrawContent(g) to render the material background and border, then ensures the inner panel is aligned via PositionMainPanelToMaterialContent.

Size changes trigger OnSizeChanged which repositions the inner panel to stay within the material content rect. DPI scaling is handled by the internal TableLayoutPanel's AutoSize and GrowAndShrink modes.

Examples

Example 1: Bind a Plain CLR Object

public class Customer
{
    public string Name { get; set; }
    public string Email { get; set; }
    public int Age { get; set; }
    public CustomerType Type { get; set; } // enum
}

// Create and bind
var recordEditor = new BeepDataRecord();
var customer = new Customer
{
    Name = "Alice",
    Email = "alice@example.com",
    Age = 30,
    Type = CustomerType.Premium
};
recordEditor.SetDataRecord(customer);

// Later, pull changes back
recordEditor.UpdateDataRecord();

Example 2: Use with BindingSource & IEditableObject

var bindingSource = new BindingSource();
bindingSource.DataSource = customersList; // List<Customer>
bindingSource.PositionChanged += (s, e) =>
{
    var current = bindingSource.Current;
    if (current != null)
        beepDataRecord1.SetDataRecord(current);
};

// Cancel edits via IEditableObject
beepDataRecord1.CancelEdit();  // rolls back to BeginEdit snapshot
beepDataRecord1.EndEdit();     // commits the edit

// Undo a single field
beepDataRecord1.UndefField("Email");

Example 3: Configure Layout with Field Prompts

var recordEditor = new BeepDataRecord
{
    FieldOrientation = FieldOrientation.Vertical,
    ShowFieldPrompts = true,
    FieldPrompts = new Dictionary<string, string>
    {
        { "Name", "Customer Name" },
        { "Email", "Email Address" },
        { "Age", "Age (years)" }
    }
};

recordEditor.SetDataRecord(customer);
recordEditor.FieldValueChanged += (s, e) =>
{
    Console.WriteLine($"Field '{e.PropertyName}' changed to '{e.PropertyValue}'");
};

recordEditor.RecordStatusChanged += (s, status) =>
{
    Console.WriteLine($"Record status: {status}");
};

Example 4: Validate and Save

beepDataRecord1.RecordValidated += (s, isValid) =>
{
    saveButton.Enabled = isValid;
};

beepDataRecord1.FieldValueChanged += (s, e) =>
{
    beepDataRecord1.UpdateDataRecord();
    // DataRecord now contains the latest values
};

// Check state before saving
if (beepDataRecord1.IsDirty)
{
    beepDataRecord1.EndEdit();
    // persist DataRecord...
    beepDataRecord1.AutoSave = true; // signal flag
}

Architecture

BeepDataRecord is a single-file control. It does not use partial classes. The layout is managed by a two-level container hierarchy:

MemberTypeResponsibility
_mainPanelPanelOuter container sized to the material content rect. Ensures fields render within themed borders.
_tableLayoutTableLayoutPanelGrid layout engine. Reconfigured by SetupHorizontalLayout / SetupVerticalLayout / SetupEmptyLayout.
_fieldsList<IBeepUIComponent>Auto-generated field component list. Each item implements IBeepUIComponent with BoundProperty, SetValue, GetValue, and OnValueChanged.
_originalDataRecordobjectDeep-copy snapshot for full undo. Created via EntityHelper.DeepCopyUsingSerialize.
_originalValuesDictionary<string, object>Per-property original value map for field-level undo via UndoField.
_currentRowTypeTypeCached type of the current data record. Used to decide whether to regenerate fields (type changed) or simply rebind (same type).

Field generation uses ControlExtensions.CreateFieldBasedOnCategory, which returns the appropriate control type based on the property's CLR type and optional metadata. Enums are automatically populated with a BindingList<SimpleItem> containing all enum values.
