Beep Docking Agent Instructions
================================

Current state: DO NOT assume BeepDockspace design-time interaction is fixed.

The user-reported blockers are:
1. BeepDockspace header/tabs are still not reliably clickable/selectable at design time.
2. Clicking a dockspace tab/header should select and activate the corresponding DockPanel.
3. Dragging from a dockspace tab/header at design time should move/stack/drop panels like a normal docking system.
4. BeepDockspace.DockPosition is not behaving correctly in layout/design-time scenarios.
5. DockPanel must NOT draw or own docked headers. The dockspace owns the header/tabs.

Architecture rule for Beep:
- BeepDockspace must be the owner of the docked header/tab strip.
- DockPanel is a page/content control only when docked in BeepDockspace.
- Do not put docked header drawing or docked header hit-testing back into DockPanel.
- Do not rely on DockPanelDesigner to receive dockspace-header clicks. DockPanel is laid out below the header, so that route is wrong.
- If a separate helper control is used for header hit testing, it must be owned internally by BeepDockspace and must forward all behavior to BeepDockspace. It must not become a separate serialized public header control unless the design-time architecture is deliberately changed.

Design-time component creation rules:
- When a design-time component/control must be serialized by Visual Studio, create it through IDesignerHost.CreateComponent.
- Do not create serialized child controls only in constructors.
- Use IComponentChangeService around Controls collection and property changes so designer.cs updates and undo/redo work.
- BeepDockingDesignerWiring is the main place to inspect for CreateComponent, Controls.Add, MovePanel, StackPanel, DropPanelAt, and ResizeDockspace behavior.

Known bad assumptions from prior work:
- It is wrong to assume a DockPanel designer can handle clicks on the dockspace header. Panels are below the header.
- It is not enough for BeepDockspace to paint tabs. The header area must also receive design-time mouse input.
- It is not enough for runtime OnMouseDown to work. Visual Studio designer can intercept mouse input before the control sees it.
- It is not enough to add IMessageFilter without proving it receives the correct HWND/lParam coordinates in the out-of-process WinForms designer.
- A private header hit-surface control inside BeepDockspace may help, but its design-time visibility and message routing must be verified in Visual Studio.

Required behavior to implement/verify:
1. With a form open in the WinForms designer, click each visible dockspace tab/header.
2. The corresponding DockPanel becomes the active visible panel.
3. The corresponding DockPanel is selected in the designer selection service.
4. Dragging a tab/header starts a design-time dock operation and can stack/move/drop the panel.
5. Dockspace resize works from the designer.
6. Dockspace.DockPosition changes layout and serialization correctly.
7. Runtime tab click/close/pin/dropdown/double-click still works.
8. DockPanel remains content-only for docked panels.

DockPosition work required:
- Inspect BeepDockingDesignerWiring.GetOrCreateDockspace, AddDockspaceToHostForm, ResizeDockspace, MovePanel, StackPanel, DropPanelAt.
- Verify DockPosition property changes call layout refresh and update dockspace bounds.
- Verify dockspace controls are actually docked/anchored/positioned according to DockPosition, not just storing the enum.
- Verify left/right/top/bottom/fill do not overlap each other or hide headers.

Icon/theming rule:
- Docking header icons should be painted through DockingCaptionPainter, which uses StyledImagePainter and SvgsUIcons.
- Icon color must be readable against the current header/tab background.
- Do not replace SVG/path-based painting with hardcoded bitmap dependencies.
