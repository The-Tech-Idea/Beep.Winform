## North-Star Goal
> Make the Beep menubar + context-menu behave like a commercial Win32/WinUI menu surface: predictable click-toggle semantics, instant hover-swap, no dismissal flicker, accessible keyboard navigation, painter-driven theming, and a clean partial-class architecture that mirrors the rest of the Beep ecosystem.

## Reported Defect (drives Phase 01)
> "I click and show menu and then I move away — it closes then opens again."
- **Root cause** (confirmed): `ContextMenuManager.ClickOutsideFilter.PreFilterMessage` calls `CloseAllMenus()` on outside `WM_LBUTTONDOWN` but **returns `false`** (line 149 of `ContextMenuManager.cs`). The dismissing click therefore continues to the message pump and re-fires `BeepMenuBar.OnMouseClick` against the originating menubar item, which calls `ShowMenuItemPopup` again. The popup closes and instantly re-opens.
- Confirmed reproduction modes:
  1. Click menu item → click a different menu item → close+reopen
  2. Click menu item → click empty area / outside → close+reopen
  3. Click menu item → mouse-only move away → close (`Deactivate` triggered) → click anywhere → reopen
  4. Click menu item → click the SAME item to toggle → close+reopen
- Fix lives in Phase 01 with a defensive partner check in `BeepMenuBar` (Phase 04 lays the long-term commercial-UX foundation).

## Established Styling Stack (Authoritative — DO NOT introduce parallel abstractions)
The Beep ecosystem already defines exactly how a control is themed and rendered. Every phase of this program **must** flow through this stack, not invent a parallel framework.

| Concern | Authoritative API | Notes |
|---|---|---|
| Per-control visual variant | `BeepControlStyle` enum + `BaseControl.ControlStyle` property | Drives Material, Fluent, GNOME, KDE, Tokyo, Neumorphism, iOS15, etc. |
| Form-level chrome contract | `FormStyle` enum + `BaseControl.UseFormStylePaint` | Controls inherit form-shell visual policy. |
| Background painting | `BackgroundPainterFactory` (per-style implementations) | Called via `BeepStyling.PaintControl(...)`. |
| Border painting | `BorderPainterFactory` (per-style implementations) | Same pipeline. |
| Shadow painting | `ShadowPainterFactory` (per-style implementations) | Same pipeline. |
| Theme colour tokens | `IBeepTheme` via `BeepThemesManager.CurrentTheme` | All colours read from theme; never hard-coded. |
| Theme fonts | `BeepFontManager` + `_currentTheme.*Font` | Already centralised. |
| List/item rendering (selection, hover, keyboard, accessibility) | `BeepListBox` (`BindingList<SimpleItem>`) | Same item model as `BeepMenuBar.MenuItems` — share the substrate. |
| Per-control orchestration | `partial class` split (Properties / Layout / Drawing / Events / Methods / Theming) + small helpers in a `Helpers/` subfolder | Mirrors `BeepListBox`, `BeepContextMenu`, every other Beep control. |

**Implication for this program**: The Beep design rule is *one rendering path per chrome concern routed through factory + style + theme*. There is **no need for a per-control painter framework**. `BeepMenuBar` already calls `BeepStyling.PaintControl(...)` correctly; the work is to clean up stale parallel infrastructure, decompose the file, and harden the popup lifecycle — not to add another painter abstraction.
